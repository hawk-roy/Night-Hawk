# 面试讲解稿

## 1 分钟项目介绍

go-order-service 是我用 Go + Gin 实现的一个订单后端服务，主要模拟电商订单的核心链路。项目包含用户注册登录、JWT 鉴权、商品查询、订单创建、库存扣减、Redis 幂等防重复下单，以及模拟支付状态流转。

这个项目的重点不是普通 CRUD，而是围绕后端真实业务问题来做：创建订单时如何保证库存和订单数据一致，用户重复点击时如何防止重复下单，支付成功或失败后订单状态如何流转，以及如何通过 Docker、测试和 CI 保证项目可运行、可验证。

## 订单创建链路

客户端创建订单时必须携带 JWT 和 `Idempotency-Key`。请求进入服务后，先经过 JWT 中间件校验身份，然后订单 handler 校验请求参数和幂等 key。

幂等部分使用 Redis `SET NX EX`。如果同一个用户使用同一个 `Idempotency-Key` 重复请求，Redis key 已存在，服务直接返回 `409`，避免重复进入下单流程。

第一次请求通过后，服务进入 MySQL 事务。在事务中查询商品和库存，并使用 `SELECT ... FOR UPDATE` 锁定库存行。库存足够时扣减库存，再写入 `orders` 和 `order_items`，最后提交事务。提交成功后，将 Redis 幂等 key 标记为 `SUCCESS:{order_no}`。

## Redis 幂等设计

Redis 幂等主要解决的是“重复请求进入下单流程”的问题。

- 为什么需要幂等：用户可能重复点击按钮，前端可能重试，请求也可能因为网络问题重复发送
- key 为什么包含 `userID`：避免不同用户使用同一个幂等 key 时互相影响
- `SET NX EX` 的作用：只有第一次请求能成功写入，后续重复请求会被拦截
- 成功后为什么 `MarkSuccess`：说明这次请求已经真正创建了订单，后续重复请求应视为已完成
- 失败后为什么 `Release`：如果事务失败或校验失败，要释放幂等占位，避免把一次失败请求永久锁死

## MySQL 事务和库存一致性

MySQL 事务解决的是“订单创建内部的一致性”问题。

- 为什么不能先扣库存再单独写订单：如果中途失败，就会出现库存扣了但订单没写进去的脏数据
- 为什么用事务：把库存扣减、订单主表、订单明细放在同一个原子操作里，保证要么一起成功，要么一起回滚
- 为什么 `SELECT FOR UPDATE`：下单时会并发读取同一库存行，不加锁可能出现超卖
- 库存不足如何处理：事务直接回滚，不写订单和订单明细，也不会消耗库存

## 支付状态流转

当前项目的支付是模拟支付，但状态流转尽量贴近真实订单系统。

- 订单初始状态是 `PENDING_PAYMENT`
- 调用 `/api/v1/payments/mock` 且 `result=SUCCESS` 时，订单变为 `PAID`
- 调用 `/api/v1/payments/mock` 且 `result=FAILED` 时，订单变为 `PAYMENT_FAILED`
- 支付失败时会恢复库存，保证订单状态和库存状态一致
- 同一订单重复支付会返回 `409`

## 日志和 X-Request-ID

项目里有请求日志中间件和 `X-Request-ID`，主要是为了排查问题更方便。

- 如果客户端传了 `X-Request-ID`，服务端会透传并写回响应头
- 如果客户端没传，服务端会自动生成
- 日志中会记录 `method`、`path`、`status`、`latency`、`client_ip` 和 `request_id`
- 这样一条请求从进入服务到返回结果，可以在日志里串起来看

## 测试和 CI

测试和 CI 的作用，是让这个项目不只是“能跑”，而是“能持续验证”。

- 单元测试主要覆盖 response、JWT、AuthMiddleware、RequestLogger、Redis 幂等逻辑
- MySQL 集成测试覆盖 OrderRepository 创建订单事务和 PaymentRepository 支付状态流转
- 默认 `go test ./...` 不依赖 MySQL / Redis 外部服务，保证本地和 CI 都能快速执行
- `RUN_INTEGRATION_TESTS=1` 时才运行仓储集成测试，避免普通开发流程被外部依赖拖慢
- GitHub Actions 会自动跑单元测试和 MySQL 集成测试，作为核心链路的质量门禁

## 面试官可能追问

### 1. 为什么不用内存 map 存订单？

因为内存数据服务重启就丢失，也无法验证真实事务和索引约束。这个项目使用 MySQL 存储订单、库存和支付流水，更接近真实后端服务。

### 2. Redis 幂等和 MySQL 事务分别解决什么问题？

Redis 幂等解决重复请求进入订单创建流程的问题；MySQL 事务解决一次订单创建内部的库存扣减、订单写入、订单明细写入一致性问题。两者关注点不同，不能互相替代。

### 3. 为什么 `Idempotency-Key` 要带 `userID`？

避免不同用户使用相同 key 时互相影响。最终 Redis key 是 `userID + Idempotency-Key` 组合。

### 4. 为什么库存查询要 `SELECT FOR UPDATE`？

因为下单会读取库存后再更新库存。如果多个事务并发读取同一库存行，不加锁可能导致超卖。`SELECT ... FOR UPDATE` 可以在事务中锁定库存行，保证同一时间只有一个事务修改该行。

### 5. 支付失败为什么恢复库存？

当前项目在创建订单时直接扣减库存，所以如果支付失败，需要把这笔订单扣掉的库存加回来，保证库存数量正确。

### 6. 为什么重复支付返回 `409`？

因为订单状态已经不是 `PENDING_PAYMENT`，再次支付属于状态冲突，所以返回 `409` 更符合语义。

### 7. 为什么默认 `go test ./...` 不跑 MySQL 集成测试？

因为普通单元测试应该不依赖外部服务，保证开发者和 CI 可以快速运行。MySQL 集成测试通过 `RUN_INTEGRATION_TESTS=1` 显式开启。

### 8. 这个项目还有哪些不足？

当前支付是模拟支付，没有接入真实第三方支付；订单只支持单商品；还没有消息队列、压测、监控和 pprof，这些可以作为后续优化方向。
